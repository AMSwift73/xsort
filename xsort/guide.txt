	This package supplies sorting algorithms and methods ported to the Odin language. It may be of help to you in the reordering of arrays of numeric primitives, arrays of structures (SoA) (and similar), and with structures-of-arrays (SOA). It does not currently assist with the sorting of strings, or with non-array datatypes (maps, trees, etc.).


"How do I use and test this library?"
	Usage is simple. Do one of three things:
	1. Use the library in your project. Place the xsort folder as a sub-folder of your source directory and use (import "xsort") statements, or
	2. Place it in your user library directory (if you have set up your development environment to find it), or (if not)
	3. Place the xsort folder in <Odin dir>/shared.
		If you do this, then the file structure will be <odin dir>/shared/xsort, and your import statements will be similar to (import "shared:xsort".)

	Testing is also uncomplicated.
	All the xsort library code lives in <project source dir>/xsort. All the testing and demonstration code lives in /test. Copy one of the test files to <src> and compile with (on Windows) either 
	"odin build . -debug -out:..\xsort-d.exe" or "odin build . -o:speed -out:..\xsort.exe"
	If not on Windows, change the "\" and lose the ".exe" and that should be enough (verify this).
	All of the test suites automatically do something when run. However, to get full value out of several of them, you will want to pass parameters.
	Parameters: 1) Number of elements in the array. 2) Number of times to perform the tests. When verifying, use 1. 3) (only on test #2). Data pattern to fill the array with. and 3 or 4) Random number seed.
	See the main proc of each file for details.


Section Alpha: "Give me the skinny."

	1A. Sorting an array of numbers is as straightforward as it is in the Odin standard library. 
	xsort.sort() takes a slice and (optionally) a toggle of ascending (forward) or descending (reverse) sort, picks a reasonable sort algorithm based (primarily) on the number of elements in the array, and performs the sort.
	Both stable and unstable sorts are considered; it is assumed you want sort speed and don't care about the relative ordering of equal values.

Examples (from 1. basic, auto, and custom sorts.odin):
xsort.sort(my_slice) // We made a slice of an array previously. We (implicitly) want an ascending sort.
xsort.sort(my_array[:], .descending) // Pass our whole array as a slice. Ask for a descending sort.


Section Beta: "Stop holding my hand. I play by my own sort rules and use my own preferred algorithm."

	1B. Let's start with custom comparators (a procedure comparing the relative sort value of two inputs). They have the same format as those used with the standard library's sort.quick_sort_proc(). They are used in several procedures, starting with the pair of general-purpose custom sort calls:
	xsort.sort_stable_cmp() takes a slice and a comparator procedure and sorts the array, using a reasonable sort algorithm, based both on the number of elements in the array and also on the size of the elements. Only stable algorithms are considered. This sometimes costs speed, but preserves the relative ordering of elements having equal sorting values.
	xsort.sort_unstable_cmp() is identical, except that it considers all potential sorts, stable and unstable. Sort speed may go up.

	Comparators compare two inputs and take the form of procedures passed to these and other calls. They return 1 if the second should go after the first, -1 if it should go before, and 0 if they are of equal sort value.

Examples (from tests/1. sorts on arrays of numbers.odin, section on comparators):
	(Ex.1) Sort an array of "stock" structures, ascending (forward). Use the "type" field of the structure.
	This comparator is placed outside of a procedure call (in, say, a "library" of sort rules).
sort_stock_type_f :: proc(a, b: stock) -> int
{
	return (int(a.type > b.type) - int(a.type < b.type))
}
	Using this comparator, sort a slice of the array; allow unstable sorts.
xsort.sort_unstable_cmp(my_slice, sort_stock_type_f) 

	(Ex.2) Perform a reverse (descending) sort on an array having elements of the "stock" structure, using the "type" value. Require a stable sort. Insert the comparator into the call.
xsort.sort_stable_cmp(my_slice, 
	proc(a, b: stock) -> int { return (int(a.type < b.type) - int(a.type > b.type)) })


	1C. Whether you call the auto-selection sorts, or want something specific, it's good to know what you get when you choose your pois ... I mean algorithm.
	The use-case for each of the available options (and other details, such as a rough idea of space and memory usage) are provided in the source files, but the short version is:
	Insertion sort (xsort.insertion_sort, etc.): Stable. Usually the fastest on arrays <= 10 elements. Craps out much above this. Tiny memory, very small space usage. Go straight to insertion sort if you're sorting tiny arrays.
	Quicksort (slice.quick_sort()): Unstable. Outperformed by insertion or shell sort below about 400 elements, but beyond this is always a solid contender. Past ~250k elements, is the fastest currently available option if datatype is u32 or 64-bit. Very low memory requirements, medium disk usage.
	Shell sort (xsort.shell_sort, etc.): Unstable. A surprisingly good backup plan for arrays too large for insertion sort, yet too small to be worth paying the up-front cost of more advanced algorithms. Tiny memory, very small space usage.
	Twinsort (xsort.twinsort, etc.): Stable. Quite fast on arrays of ~40 < elements < ~1m (outperforms C++ std::stable_sort, std::sort, C qsort, and odin sort.quicksort, often by quite a lot) - if the elements are small (< ~50 bytes). Starts to slow down on arrays of large structures, but is always at least a reasonable choice if stability is required. Substantial memory requirement, medium-small space usage.
	Radix sort (xsort.radix_sort, etc.): Unstable. A more specialized option with strong benefits and downsides. Sort not by comparison, but by performing a series of reorderings on each (by default 8-bit) digit of the number, from lowest to highest place value (so, an LSD rather than MSD radix sort). Very fast on small datatypes (i8, u16, and the like), and also with large arrays of i32; very slow on large datatypes (int, u64, etc.) and on very small arrays. Radix sort can in some cases be so fast that it may be worth customizing your large dataset to make best use of it.


Section Gamma: "Enough about arrays of numbers. I have arrays of structures to sort."

	This section is demonstrated in test/3. sorts on arrays of structs.odin.
	Sorting on arrays of structs uses custom comparators. The differences from numeric arrays are several: 
	2A: The comparator must specify the structure element(s) to sort on. Also, the array type definition must be in-scope of the sort procedure call (true also for arrays of numbers, but something like a "i32" is always in-scope). 
	2B: With arrays of structs, stability now become important - and often worth paying for. A stable sort preserves the relative ordering of equal-value elements. This means that when you, say, sort an inventory by price, ids that were previously in order remain in order within each price. While a custom comparator can be tweaked to accomplish this even for an unstable sort, doing so often costs more time than it saves. Happily, the twinsort algorithm is both stable and fairly fast.
	2C: With structs, one quickly starts wishing for multi-sorts - primary, secondary, and even tertiary search terms. Test suit #2 demonstrates an efficient dual-sort.
	2D: Finally, when we deal with arrays of structs, we're now talking about larger array elements - and that means code starts to chug. Read on.


Section Delta. "I use my favorite sort. I add just a few elements to the structs in my data arrays. Performance dies."

	It's clearly time, then, for you to get a peep into the wonderful world of indexed sorting. See "tests/4. reindex, sort from index, batch sort.odin" for demonstrations.

	3A. The first step is separate the reindexing of sort order from the actual movement of elements in the array. In batched sorting, the sorting algorithm changes only an index and the latter is done all at once at the end. When element size exceeds (very roughly) 50 bytes, this may reduce the total time taken, but a sufficiently small array might never benefit.
	A call to a batch sort is simple "xsort.twinsort_cmp_batched(my_array, my_comparator_procedure)".
See "xsort.insertion_sort_cmp_batch() and friends for your options.
	The procedures "xsort.sort_cmp_stable" and "xsort.sort_cmp_unstable" automatically make use of batched indexed sorting where practicable, greatly improving their ability to cope with arrays of ever-growing structs. The standard library also makes indexed sorting available; consult "slice.sort_with_indices()" and elsewhere.

	3B. The next step is to let you issue separate reorder and sort-from-index calls. Here, your motivations are as likely to be capability-driven as performance-driven:
	You may desire to save sorts, or to issue multi-criteria sorts, or to sort every array in a struct-of-arrays according to information in one (or more) of them - or you may not want to sort that huge array at all, but to sort by index on it, or to flexibly display it by index in your user interface. In all of these case and more, you workflow will include steps similar to those demonstrated in 4. reindex, sort from index, batch sort.odin:

// Sort by id (ascending order). Initialize (reset) index first.
xsort.gen_sorted_index_stable(fruits.id^[:], fruit_sort_index[:], 
	proc(a, b: u16) -> int { return (int(a > b) - int(a < b)) },
	true)

// Sort by number. Do not overwrite previous sort (retain id order for a given number).
xsort.gen_sorted_index_stable(fruits.num^[:], fruit_sort_index[:], 
	proc(a, b: u16) -> int { return (int(a > b) - int(a < b)) },
	false)

// Print out, in sorted order without array reordering.
for i in 0 ..< nmemb_soa
{
	idx := fruit_sort_index[i]
	fmt.printfln("%-13s: num = %d, price = %.2f, deliciousness = {}", 
		fruit_names[fruits.id[idx]], fruits.num[idx], fruits.price[idx], delish_names[fruits.deliciousness[idx]])
}
